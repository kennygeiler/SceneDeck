# SceneDeck Master Plan

## Overview

SceneDeck is a searchable database of iconic cinema shots tagged with structured camera motion metadata. This plan organizes the build into 7 milestone-based feature areas, ordered by dependency and priority. The metadata overlay on video playback is the hero feature; frontend quality is explicitly prioritized over pipeline completeness (C-08).

Timeline: 1-2 weeks (~12 days). All code generated by AI agents (C-01).

---

### Milestone 1: Foundation and Design System

**Goal**: Establish the project scaffold, shared taxonomy constants, design system tokens, and dev tooling so all subsequent milestones have a stable base to build on. Database provisioning is deferred -- this milestone focuses on what the overlay and UI need to start.

**Deliverables**:
- [ ] Next.js 15 App Router project initialized with TypeScript, Tailwind CSS 4, shadcn/ui, Framer Motion
- [ ] Shared camera movement taxonomy constants defined in TypeScript (21 movement types, 15 directions, 7 speeds, 15 shot sizes, 15 angles, 6 duration categories, compound notation)
- [ ] Python pipeline directory (`/pipeline`) initialized with dependencies (PySceneDetect, google-generativeai, anthropic, httpx, python-dotenv)
- [ ] Matching taxonomy constants defined in Python, identical to TypeScript version
- [ ] Design tokens file (tokens.css) with color palette, typography scale, and spacing system based on visual direction: "technical precision meets cinematic elegance"
- [ ] Design artifact: component style guide documenting visual approach for overlay, cards, filters, and verification UI (markdown + token references)
- [ ] Site shell: root layout with header, navigation structure, dark cinematic theme applied
- [ ] Vercel project configured for deployment from GitHub

**Dependencies**: None

**Acceptance Criteria**:
- `pnpm dev` starts the Next.js app without errors
- TypeScript taxonomy enums compile and match Python taxonomy dicts exactly (same keys, same values)
- Design tokens CSS custom properties available in the app and visually applied to the site shell
- Vercel preview deployment accessible at a live URL
- Site shell renders with cinematic dark theme, not default/generic styling

**Scope Boundaries**: No database provisioning, no schema, no Drizzle ORM setup. No page content beyond the shell. No video playback.

### Milestone 2: Hero Feature -- Video Metadata Overlay

**Goal**: Build the single most important UI element: video playback with a visually striking metadata overlay that renders camera motion data (movement type, direction arrows, trajectory paths, shot size, speed) synchronized to currentTime. This is the portfolio differentiator (C-12, SC-05).

**Deliverables**:
- [ ] `VideoOverlay` component: HTML5 video element with absolutely-positioned Canvas + SVG layers
- [ ] Canvas layer renders per-frame overlays: direction arrows, motion trajectory paths, speed indicators
- [ ] SVG layer renders vector annotations: movement type label, shot size badge, angle indicator
- [ ] Overlay data synchronized to `video.currentTime` via `requestAnimationFrame` loop
- [ ] Framer Motion transitions for overlay state changes (e.g., compound movement phase transitions)
- [ ] Overlay toggle controls: show/hide individual overlay groups (movement, framing, speed)
- [ ] Overlay legend explaining visual language of arrows, badges, and annotations
- [ ] Shot detail page (`/shot/[id]`) with video player + overlay + metadata panel
- [ ] Responsive layout -- desktop-first but usable on tablet
- [ ] Screen-recordable quality: overlays render at full resolution, no blinking or desync
- [ ] Seed data: 3-5 manually created shot records with hardcoded mock data for development/demo

**Dependencies**: Milestone 1 (Foundation)

**Acceptance Criteria**:
- Visiting `/shot/[id]` plays a video clip with metadata overlay rendered on top
- Overlay arrows/paths update smoothly as video plays (no visible frame drops at 30fps+)
- Movement type, direction, speed, shot size, and angle are all visually represented
- Compound movements display sequential phase transitions
- Overlay is visually striking -- cinematic aesthetic, not academic or raw
- Works when screen-recorded (no flickering, no z-index issues)
- Overlay groups can be toggled on/off independently

**Scope Boundaries**: No database queries -- uses hardcoded mock data. No browse page, no search. No verification workflow.

### Milestone 3: Database, Browse, and Search

**Goal**: Provision the database, replace mock data with real queries, and implement directory browsing with faceted filters and semantic natural language search. Users get two complementary ways to find shots (SC-03, SC-04).

**Deliverables**:
- [ ] Neon PostgreSQL database provisioned via Vercel Marketplace with pgvector extension enabled
- [ ] Drizzle ORM configured with schema defining all tables: `films`, `shots`, `shot_metadata`, `shot_semantic`, `verifications`, `shot_embeddings` (per architecture.md)
- [ ] Drizzle migration generated and applied to Neon
- [ ] Seed script that inserts development data (films + shots + metadata) into Neon
- [ ] Shot detail page (`/shot/[id]`) refactored to query from Neon instead of mock data
- [ ] Browse page (`/browse`) with grid/list view of shot cards
- [ ] `ShotCard` component: thumbnail, film title, movement type badge, director, duration
- [ ] `FilterSidebar` component: faceted filters for film, director, movement type, shot size, angle, speed
- [ ] Filter state managed via URL search params (shareable filter URLs)
- [ ] `SearchBar` component: natural language search input
- [ ] `/api/search` route handler: accepts query string, generates embedding via OpenAI text-embedding-3-small, performs pgvector similarity search
- [ ] `/api/shots` route handler: supports filter parameters, returns paginated shot list
- [ ] Embedding generation utility: generates and stores embeddings for all shots in `shot_embeddings` table
- [ ] Landing page (`/`) with hero visual, featured shots carousel, and prominent search bar
- [ ] Designed empty states and no-match states for browse and search

**Dependencies**: Milestone 1 (Foundation), Milestone 2 (Shot detail page and overlay exist)

**Acceptance Criteria**:
- `pnpm drizzle-kit push` applies schema to Neon without errors
- `/browse` displays all shots with working filters; selecting a filter updates results without full page reload
- Searching "slow tracking shot through a hallway" returns relevant results ranked by semantic similarity
- Clicking a shot card navigates to `/shot/[id]` with overlay rendering from database data
- Landing page loads with featured shots and functional search
- Filter combinations work correctly (AND logic across filter types)
- Filter URLs are shareable -- visiting a filter URL restores the filter state

**Scope Boundaries**: No QA verification. No data export. No pipeline ingestion. Seed data is manually curated via seed script, not pipeline-generated.

### Milestone 4: QA Verification System

**Goal**: Implement the human QA verification workflow where users rate AI-generated metadata accuracy on a 0-5 scale per shot. This is the confidence metric -- there is no separate AI confidence score (KD-05, SC-06).

**Deliverables**:
- [ ] Verification page (`/verify`) listing shots needing verification, sorted by unverified first
- [ ] `VerificationPanel` component: displays current metadata alongside video playback with overlay, 0-5 star rating UI, per-field accuracy toggles (movement_type, direction, speed, etc.)
- [ ] Optional corrections: user can submit corrected values for any metadata field
- [ ] `/api/verify` route handler: POST creates/updates verification record, PATCH updates shot_metadata if corrections provided
- [ ] Verification status badge on shot cards in browse view (unverified, verified with rating)
- [ ] Classification source badge visible on shot detail and verification pages ('gemini', 'raft', 'manual')
- [ ] Public access -- no authentication required (C-07)
- [ ] Zod validation schema for verification submissions

**Dependencies**: Milestone 3 (Database and browse page must exist)

**Acceptance Criteria**:
- User can visit `/verify`, select a shot, watch it with overlay, and submit a 0-5 rating
- Per-field ratings are stored in `verifications.field_ratings` JSONB
- Corrections update `shot_metadata` values and set `classification_source` to 'manual'
- Verification status visible on browse page shot cards
- Multiple verifications per shot are stored (historical record)
- Verification workflow completable in under 1 minute per shot
- Classification source is visible on shot detail and verification pages

**Scope Boundaries**: No batch verification. No automated verification triggers. No verification analytics/dashboard.

### Milestone 5: Data Pipeline -- Gemini Classification

**Goal**: Build the offline Python pipeline that ingests video files, detects shot boundaries, classifies camera motion via Gemini 2.0 Flash, and uploads results to Neon + Vercel Blob. Validate Gemini accuracy on 10-20 clips before scaling (C-03).

**Deliverables**:
- [ ] Ingest module: accepts source video file, extracts metadata (film title, director, year), validates format with FFmpeg/ffprobe
- [ ] Shot detection module: PySceneDetect AdaptiveDetector identifies shot boundaries, FFmpeg extracts individual clips
- [ ] Gemini classification module: uploads clip to Google Files API, sends structured prompt requesting taxonomy JSON, parses response into shot_metadata schema
- [ ] Thumbnail extraction: FFmpeg generates midpoint thumbnail for each shot
- [ ] Upload module: writes shot metadata to Neon via direct SQL/httpx, uploads clips + thumbnails to Vercel Blob
- [ ] TMDB enrichment (optional): fetch film metadata from TMDB API to populate tmdb_id and supplement film records
- [ ] Accuracy validation checkpoint: script to run Gemini on 10-20 clips, output classification results for human review before full batch processing
- [ ] CLI entry point: `python pipeline/main.py --input video.mp4 --film "Title" --director "Name" --year 2000`
- [ ] Pipeline configuration file for API keys and Neon/Blob connection strings
- [ ] Requirements.txt with pinned dependencies

**Dependencies**: Milestone 1 (Foundation -- taxonomy constants must exist), Milestone 3 (Database schema must exist for uploads)

**Acceptance Criteria**:
- Running the pipeline on a single source video file produces: shot clips in Vercel Blob, thumbnails in Vercel Blob, shot records in Neon with Tier 1 metadata
- Gemini classification returns valid taxonomy values for all 21 movement types
- Accuracy validation on 10-20 clips shows results for human review (pass/fail threshold determined by operator)
- Pipeline handles compound movements (multi-phase shots)
- At least one complete scene ingested end-to-end (SC-08)
- Pipeline-ingested shots appear correctly in the Next.js browse and detail views

**Scope Boundaries**: No RAFT/Modal GPU fallback pipeline (C-03 -- build only if Gemini accuracy is below threshold after validation). No real-time ingestion. No scene grouping. No Tier 2 semantic metadata extraction.

---

### Milestone 6: Data Export

**Goal**: Provide data export capabilities so shot metadata can be consumed by external tools and workflows (SC-07, C-13).

**Deliverables**:
- [ ] `/api/export` route handler: accepts format parameter (json, csv), optional filter parameters
- [ ] JSON export: returns full taxonomy metadata for filtered or all shots as structured JSON
- [ ] CSV export: returns flat CSV with one row per shot, columns for all Tier 1 metadata fields
- [ ] `ExportDialog` component: UI for selecting format, applying filters, triggering download
- [ ] Export accessible from browse page and shot detail page

**Dependencies**: Milestone 3 (Database and browse page for export trigger UI)

**Acceptance Criteria**:
- Downloading JSON export produces valid JSON containing all shot metadata
- Downloading CSV export produces valid CSV openable in Excel/Google Sheets
- Filters applied on browse page carry through to export (export what you see)
- Export includes all Tier 1 fields: movement_type, direction, speed, shot_size, angles, duration_cat, compound_parts

**Scope Boundaries**: No MCP plugin. No API key authentication for export. No streaming export for large datasets.

---

### Milestone 7: Polish, Seed Data, and Deploy

**Goal**: Final visual polish, seed dataset population, performance optimization, and production deployment. Ensure the live demo creates a strong first impression for portfolio evaluators.

**Deliverables**:
- [ ] Seed dataset: 50-100 iconic art house cinema shots processed through pipeline and verified via QA system
- [ ] Landing page polish: hero visual moment, smooth page transitions, loading states with skeleton screens
- [ ] Responsive refinements: ensure browse, search, overlay, and verification work on common screen sizes
- [ ] Performance: video preloading, image optimization (next/image for thumbnails), metadata query optimization
- [ ] Error handling: graceful fallbacks for missing videos, failed searches, empty states across all pages
- [ ] Loading and error states designed (not default browser states)
- [ ] SEO: meta tags, Open Graph images for social sharing of individual shots
- [ ] Production Vercel deployment with environment variables configured
- [ ] Vercel Blob CDN verified for video delivery performance
- [ ] README with setup instructions and .env.example
- [ ] Demo script documenting the recommended walkthrough path for portfolio evaluators

**Dependencies**: All previous milestones

**Acceptance Criteria**:
- Live URL loads without errors (SC-01)
- 50-100 shots with Tier 1 metadata, all human-verified (SC-02)
- First-visit experience: portfolio evaluator can interact meaningfully within 30 seconds
- No console errors, no broken images, no dead links
- Page load time under 3 seconds on broadband
- Hero overlay is screen-recordable for social media demos
- All user-facing states (loading, empty, error, no-match) are designed, not raw

**Scope Boundaries**: No Tier 2/3 metadata. No MCP plugin. No ComfyUI integration. No article series (post-launch).

## Priority Order (if time-constrained)

Per C-08 (frontend quality over pipeline completeness), if the timeline compresses:

1. **Must complete:** M1 (Foundation), M2 (Hero Overlay), M3 (Database, Browse & Search)
2. **Should complete:** M4 (QA Verification), M5 (Pipeline -- at least validation checkpoint)
3. **Complete if time allows:** M6 (Export), M7 (full seed data population and polish)

The frontend must be visually impressive even with manually entered seed data. The pipeline can degrade to semi-manual without compromising the demo.
