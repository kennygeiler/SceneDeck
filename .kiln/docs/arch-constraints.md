# SceneDeck Architectural Constraints

These are hard constraints that all planners and implementers must respect. Violating any of these requires an ADR and operator approval.

## C-01: Zero Manual Coding

Every line of application code must be generated by AI coding agents. The operator may only: prompt, approve, add API keys, permission services, do work relays, update environment variables. This applies to both the Next.js web app and the Python pipeline.

**Implication for planners**: All code must be writable by Claude/Cursor in a single prompt or short conversation. Avoid patterns that require deep multi-file coordination that AI agents struggle with (e.g., complex state machines spanning 10+ files).

## C-02: Fixed Camera Movement Taxonomy

The taxonomy is hardcoded and immutable at runtime:
- 21 movement types (static, pan, tilt, dolly, truck, pedestal, crane, boom, zoom, dolly_zoom, handheld, steadicam, drone, aerial, arc, whip_pan, whip_tilt, rack_focus, follow, reveal, reframe)
- 15 direction values
- 7 speed categories
- 15 shot sizes
- 15 camera angles (6 vertical + 5 horizontal + 4 special)
- 6 duration categories
- Compound notation: max 3 simultaneous components as ordered `{type}:{direction}` pairs

These must be defined as shared constants (TypeScript enums/const objects in web app, Python enums/dicts in pipeline). A single source-of-truth taxonomy file must exist; both codebases reference it or duplicate it identically.

## C-03: Gemini-First Classification Pipeline

Camera motion classification MUST use Gemini 2.0 Flash as the primary classifier. The RAFT/Modal GPU pipeline is a fallback only, activated if Gemini accuracy is below acceptable threshold after human QA review of the first 10-20 clips. Do not build the RAFT pipeline upfront -- build the Gemini path first and validate.

**Implication for planners**: The pipeline milestone must include a Gemini validation checkpoint before any RAFT work is planned.

## C-04: Neon PostgreSQL as Sole Database

All persistent data lives in a single Neon PostgreSQL instance (free tier: 0.5 GB storage, 100 hours compute/month). No Redis, no separate search service, no secondary databases. Semantic search uses pgvector extension within Neon.

**Implication for planners**: Schema must be efficient. Total data for 100 shots with full metadata + embeddings must fit within 0.5 GB. Estimated: ~50 MB (well within limits).

## C-05: Vercel Blob for All Media

All video clips and thumbnails are stored on Vercel Blob. No S3, no Cloudflare R2, no self-hosted media. Videos served directly from Blob CDN URLs to the browser.

**Implication for planners**: Upload pipeline must use the Vercel Blob SDK or REST API. Maximum file size on Vercel Blob free tier is 500 MB per file (more than sufficient for shot clips).

## C-06: App Router Only (No Pages Router)

The Next.js app must use the App Router exclusively. No Pages Router, no `getServerSideProps`, no `getStaticProps`. Use Server Components, Server Actions, and Route Handlers.

**Implication for planners**: All data fetching happens in Server Components or Route Handlers. Client Components are used only for interactive elements (video player, overlays, search input, verification form).

## C-07: No Authentication in v1

No user accounts, no login, no session management, no access controls. The QA verification system is open -- anyone can rate shots. This is a portfolio demo, not a multi-tenant product.

**Implication for planners**: Do not plan auth middleware, user tables, or protected routes. The verify endpoint is public.

## C-08: Frontend Quality Over Pipeline Completeness

If time runs short, the UI must be visually impressive even if the data pipeline requires manual correction. The metadata overlay on video playback is the single most important UI element.

**Implication for planners**: Frontend milestones must be prioritized earlier than pipeline automation milestones. The pipeline can degrade to semi-manual (operator runs scripts, corrects via QA UI) without compromising the demo.

## C-09: 50-100 Shots Maximum for v1

The seed dataset is 50-100 curated iconic shots from art house cinema. The architecture does not need to handle thousands of shots, high concurrency, or streaming ingestion. Design for small-scale correctness, not scale.

**Implication for planners**: No pagination optimization, no background job queues, no caching layers. Simple queries and full-page data loads are fine.

## C-10: Pipeline Runs Offline (Not on Vercel)

The Python data pipeline runs on the operator's local machine or in GitHub Actions. It is not deployed as a Vercel serverless function or edge function. It writes directly to Neon and Vercel Blob via their APIs.

**Implication for planners**: Pipeline code lives in a separate directory (e.g., `/pipeline`) with its own Python dependencies. It shares taxonomy constants with the web app but is otherwise independent.

## C-11: 1-2 Week Timeline

Total build time budget is 1-2 weeks of full-time work. Every architectural choice must optimize for speed of implementation and vibe-code compatibility, not scalability or future-proofing.

**Implication for planners**: Milestones must be small (2-3 days max). Features that take more than a day to implement should be questioned for scope.

## C-12: Metadata Overlay is the Hero Feature

The video playback with metadata overlay (movement type, direction arrows, trajectory visualization, shot size, speed) is the primary differentiator and must receive the most design and engineering attention. It must be:
- Visually striking (not academic or raw)
- Screen-recordable (for social media demos)
- Synchronized to video currentTime
- Rendered via HTML5 Canvas/SVG positioned over the video element

## C-13: Data Export Required

At least one export format (JSON or CSV) must be available for shot metadata. This is a success criterion (SC-07). The export endpoint returns the full taxonomy metadata for filtered or all shots.

## C-14: Shared Taxonomy Constants

The camera movement taxonomy must be defined identically in both the TypeScript web app and the Python pipeline. Any mismatch between the two is a bug. Planners should ensure a single milestone includes defining the taxonomy in both codebases with validation.
